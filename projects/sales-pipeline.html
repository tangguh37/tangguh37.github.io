Problem

How do you stream 186,000 rows of transactional sales data from a PostgreSQL database into an analytics warehouse in real-time — without writing complex ETL scripts or managing infrastructure?

This project demonstrates a modern CDC (Change Data Capture) pipeline: Neon (serverless PostgreSQL) → Streamkap (CDC engine) → MotherDuck (cloud DuckDB) → Power BI dashboard.

Tech Stack

  • Neon — Serverless PostgreSQL as source OLTP database with logical replication
  • Streamkap — CDC engine reading via publication, writing via JDBC sink
  • MotherDuck — Cloud DuckDB with raw (streamkap) and analytics schemas
  • DuckDB — Analytical query engine powering aggregated views
  • Power BI — BI tool connected via MotherDuck's PostgreSQL endpoint
  • Kaggle Dataset — 2019 electronics store sales data

What I Built

Configured Neon with a dedicated publication (streamkap_pub) for logical replication. Set up Streamkap source connector pointing to Neon and destination connector writing to MotherDuck via JDBC. Designed MotherDuck schemas: streamkap (raw CDC data) and analytics with aggregated views including monthly sales trends, top products, daily sales, and geographic breakdowns by city. Connected Power BI using DirectQuery for live dashboarding.

Dashboard Preview

Power BI dashboard: real-time sales metrics, monthly trends, top products, and city-wise breakdown

Learnings

This was my first end-to-end CDC pipeline using the modern data stack. Key takeaways: logical replication requires careful publication configuration, CDC latency is highly dependent on network and batch settings, and MotherDuck's PG-compatible endpoint makes it trivial to connect BI tools without custom connectors.
